## 1.0 Core Workflow
*Atomic input → process → output for the Afloat application unit.*

### 1.1 Input
*   API request with valid auth header (JWT via `jose`).
*   Request body: user message + optional client-echoed history.

### 1.2 Process
*   Auth check → Session state enforcement → Stream processing → Response.
*   Session limits: `MAX_LLM_CALLS = 2`, `MAX_DURATION_MS = 120000`.
*   History: client-echoed, sanitized (last 4 entries, role/content validated, 2000-char truncation).
*   History is ephemeral — never persisted to the data layer.

### 1.3 Output
*   **JSON responses:** Health, Session Start/End, Message.
*   **ZIP export:** `application/zip` container with `data.json` + `data.csv` (portable format).
*   **Default export:** `application/json` when no format is specified (codec fallback).

### 1.4 Validation Gate
*   **Command:** `npx vitest run`
*   **Requirement:** 107/107 tests (8 files). 0 lint errors. Clean `next build`.

## 2.0 Session Lifecycle & Ownership
*State machine for session and provenance authorization.*

### 2.1 Transient State (Ephemeral Stream)
*   History accepted from client, sanitized, used for LLM call, then discarded (REQ-A5).
*   No conversation history written to Redis — forward secrecy analog.

### 2.2 Persistent State (Channel Ownership)
*   DPR (Decision Provenance Record) chain proves ownership after session deletion (REQ-B2/B5).
*   Chain is HMAC-signed and hash-linked — tamper detection via recomputation.

### 2.3 State Transitions
*   **Auth validation:** JWT token required for all API access (REQ-B1/B4).
*   **Session-user parity:** `session.user_id` must match authenticated user (REQ-B6a/b).
*   **Post-deletion fallback:** If session key is gone, DPR chain `actor_id` is checked.

### 2.4 Failure Modes
*   **401** — Missing or invalid auth header.
*   **403** — No active subscription, foreign session claim, or foreign DPR chain.
*   **404** — Session or chain does not exist.
*   **409** — Session turns exhausted or timed out.

## 3.0 Security & Integrity
*Hard constraints verified by test probes.*

### 3.1 Auth Enforcement
*   401 without auth header.
*   403 without active subscription.
*   403 when accessing another user's session (live or deleted).

### 3.2 Stream Integrity
*   **Sanitization (REQ-A2):** System-role entries and null/malformed entries stripped from client history.
*   **Window truncation (REQ-A3):** Only last 4 history entries kept; content truncated at 2000 chars.
*   **Empty normalization (REQ-A4):** `[]`, `undefined`, and absent history key all normalize to empty array.

### 3.3 Export Integrity
*   **Magic bytes (REQ-C2):** ZIP container starts with `PK\x03\x04` signature.
*   **Directory (REQ-C3):** ZIP contains exactly `data.json` + `data.csv`.
*   **Headers (REQ-C1):** `Content-Type: application/zip` + `Content-Disposition: attachment`.
*   **Codec fallback (REQ-C4):** Default format returns parseable JSON.
*   **Empty payload (REQ-C5):** Zero sessions still produces valid ZIP with empty data.

## 4.0 Rate Limits & Exhaustion
*Per-session capacity — no cross-session concurrency enforcement exists.*

### 4.1 Per-Session Limits
*   **Turns:** `MAX_LLM_CALLS = 2` (configurable constant).
*   **Duration:** `MAX_DURATION_MS = 120000` (2 minutes).

### 4.2 Exhaustion Handling
*   Session turns exhausted → **409 Conflict** → End session → Start new.
*   Session timed out → **409 Conflict** → Same recovery path.
*   Specific error codes (409/403) returned — no silent failures.
